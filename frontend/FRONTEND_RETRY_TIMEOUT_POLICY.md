# Frontend Retry / Timeout Policy

## Goal
TICKETEER 프론트는 모든 요청을 무조건 재시도하지 않고,
**요청 성격에 따라 timeout과 retry 정책을 다르게 적용**합니다.

핵심 기준은 다음과 같습니다.

- 조회성 요청은 제한적으로 재시도 가능
- 상태 변경 요청은 자동 재시도를 매우 신중하게 적용
- 예매 확정처럼 중복 실행 위험이 큰 요청은 자동 재시도하지 않음

---

## 1. Why This Matters

티켓팅 UI에서는 네트워크가 느리거나 일시적으로 실패할 수 있습니다.

하지만 모든 요청에 자동 재시도를 걸면 오히려 문제가 생길 수 있습니다.

예:
- 좌석 조회는 다시 시도해도 비교적 안전
- 예매 확정은 자동 재시도 시 중복 행동처럼 느껴질 수 있음
- 사용자는 같은 버튼을 여러 번 누르고 싶어질 수 있음

따라서 요청 종류별 정책 분리가 필요합니다.

---

## 2. Current Policy

### `fetchSeats`
- 성격: 조회성 GET
- timeout: 4초
- retry: 1회

이유:
- 좌석 목록은 서버 상태 재동기화에 중요
- 일시적인 네트워크 흔들림이면 1회 재시도가 UX에 유리
- 중복 실행 부작용이 적음

### `getQueueStatus`
- 성격: 조회성 GET + polling
- timeout: 3초
- retry: 1회

이유:
- polling 중 짧은 네트워크 문제로 바로 실패 처리하면 UX가 불안정해짐
- 다만 과도한 재시도는 polling과 겹쳐 불필요한 요청 증가를 만들 수 있어 1회만 허용

### `enterQueue`
- 성격: 상태 생성 POST
- timeout: 5초
- retry: 없음

이유:
- 대기열 진입은 생성성 요청이라 자동 재시도 시 중복 진입 UX가 꼬일 수 있음
- 이미 프론트에서 중복 진입 방지 로직이 있음

### `holdSeat`
- 성격: 상태 변경 POST
- timeout: 5초
- retry: 없음

이유:
- 좌석 홀드는 경쟁 상황과 직접 연결됨
- 자동 재시도 시 사용자가 의도하지 않은 추가 요청처럼 보일 수 있음
- 실패 시 좌석 재조회로 상태를 다시 맞추는 것이 더 안전

### `createBooking`
- 성격: 최종 확정 POST
- timeout: 5초
- retry: 없음

이유:
- 예매 확정은 가장 민감한 요청
- 자동 재시도는 중복 실행 불안감을 줄 수 있음
- 예매 완료 여부는 서버 응답과 결과 UI로만 확정하는 방향이 안전

---

## 3. Why We Avoid Auto Retry for Mutations

mutation 요청은 단순 실패보다 **중복 실행 오해**가 더 위험합니다.

특히 티켓팅에서는 사용자가 이런 의심을 하게 되면 UX 신뢰가 크게 떨어집니다.

- “결제가 두 번 된 건 아닌가?”
- “예매 요청이 중복으로 들어간 건 아닌가?”
- “좌석이 이미 잡혔는데 또 요청된 건 아닌가?”

그래서 현재 정책은:
- mutation은 자동 재시도하지 않음
- 대신 버튼 중복 클릭 방지
- 실패 시 사용자 메시지 표시
- 좌석 재조회로 서버 상태 재동기화

를 선택했습니다.

---

## 4. Timeout Strategy

timeout은 너무 짧아도 UX를 망치고,
너무 길어도 사용자가 멈춘 화면처럼 느끼게 만듭니다.

현재 기준:
- queue status: 3초
- seats fetch: 4초
- mutation 계열: 5초

이 기준은
- polling 요청은 더 짧게
- 사용자 액션 기반 요청은 조금 더 여유 있게

잡은 것입니다.

---

## 5. Summary

현재 TICKETEER 프론트의 retry / timeout 정책은 다음과 같습니다.

- 조회 요청은 짧은 timeout + 제한적 재시도
- mutation 요청은 timeout만 적용하고 자동 재시도하지 않음
- 예매 확정은 자동 재시도보다 중복 요청 방지와 결과 표시를 우선

이 정책은
**티켓팅 도메인에서 중요한 사용자 신뢰와 상태 정합성**을 고려한 선택입니다.
