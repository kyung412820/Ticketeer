즉 이 문서는 “무엇을 만들었는가”보다,

## 2. 프로젝트 한 줄 정의

TICKETEER는 **대기열 기반 티켓 예매 시스템**으로,  
동일 좌석에 대한 **중복 홀드 / 중복 예매 방지**와  
실시간 상태 변화가 많은 예매 흐름에서의 **화면 정합성 유지**를 핵심 문제로 두고 구현한 프로젝트다.

---

## 3. 왜 이 프로젝트를 만들었는가

티켓팅 서비스는 겉보기에는 단순한 예매 서비스처럼 보이지만, 실제로는 일반 CRUD보다 훨씬 까다로운 문제를 가진다.

대표적으로:

- 많은 사용자가 짧은 시간 동안 동시에 진입한다.
- 여러 사용자가 같은 좌석을 동시에 선택한다.
- 임시 점유와 최종 확정의 성격이 다르다.
- 프론트 화면이 서버 상태보다 늦게 반영되면 잘못된 안내를 줄 수 있다.
- “예매가 정말 완료됐는가”, “중복 요청은 없었는가”, “화면이 거짓말하지 않는가”가 서비스 신뢰와 직결된다.

즉 이 프로젝트는 단순히 “예매 기능 만들기”가 아니라,

1. 백엔드에서는  
   **동시 요청 환경에서도 데이터 정합성을 보장하는가**

2. 프론트에서는  
   **실시간 상태 변화 속에서 사용자에게 신뢰 가능한 UI를 제공하는가**

를 해결하기 위해 시작했다.

---

## 4. 핵심 사용자 흐름

TICKETEER의 핵심 흐름은 다음과 같다.

1. 공연 목록 조회
2. 공연 상세 조회
3. 대기열 진입
4. 대기열 상태 조회
5. 좌석 목록 조회
6. 좌석 홀드
7. 예매 확정

이 흐름을 설계할 때 가장 중요했던 점은,  
**모든 단계가 이어져야 하는 것이 아니라, 경쟁 상황에서도 안전해야 한다는 점**이었다.

## 5. 기술 스택과 선택 이유

## 5-1. Frontend
- Next.js
- TypeScript
- Tailwind CSS
- React Query
- Playwright

### 왜 Next.js를 사용했는가
Next.js는 React 기반이면서도 **SSR/CSR를 페이지 성격에 따라 나눠 설계하기 좋다**는 장점이 있다.

이 프로젝트에서는:
- `/events`, `/events/[id]`는 조회 중심 화면
- `/queue/[eventId]`, `/booking/[eventId]`는 실시간 상호작용 중심 화면

으로 성격이 다르기 때문에,  
단순 React SPA보다 **렌더링 전략을 설명하기 좋은 구조**가 필요했다.

Next.js를 선택한 이유는:
- 서버 중심 렌더링과 클라이언트 중심 렌더링을 구분하기 쉬움
- 페이지 기반 구조가 명확함
- 포트폴리오에서 “렌더링 전략을 고민했다”는 설명이 가능함

### 왜 TypeScript를 사용했는가
예매 흐름은 API 응답 구조가 많고 상태가 자주 바뀐다.  
이때 타입이 없으면 다음 문제가 생긴다.

- 응답 구조 오해
- nullable 상태 누락
- 상태 전환 시 실수 증가
- 리팩터링 시 안정성 저하

TypeScript를 사용함으로써:
- API 응답 타입 일관화
- 상태 전환 로직 안정화
- 컴포넌트 props 구조 명확화

를 기대할 수 있었다.

### 왜 Tailwind CSS를 사용했는가
이 프로젝트의 목적은 디자이너 수준의 비주얼 작업이 아니라,  
**빠르게 일관된 UI를 만들고 상태별 시각 차이를 분명하게 드러내는 것**이었다.

Tailwind를 선택한 이유:
- 빠른 개발 속도
- disabled / focus / status 구분이 쉬움
- 시각적 일관성 유지에 유리함

### 왜 React Query를 도입했는가
초기에는 로컬 state + fetch wrapper로도 충분했지만,  
대기열 상태와 좌석 목록은 명확하게 **서버 상태(server state)**였다.

특히:
- queue status는 polling 필요
- seats list는 hold / booking 후 재동기화 필요
- stale state를 줄여야 함

따라서 React Query를 도입해:
- queue status polling 관리
- seats list query 관리
- invalidate 기반 재조회
- 서버 상태와 UI 상태의 역할 분리

를 명확하게 했다.

### 왜 Playwright를 사용했는가
프론트 메인 포트폴리오에서는 “화면이 보인다”보다  
**실제 브라우저 기준으로 핵심 흐름이 검증된다**는 점이 중요하다.

Playwright를 선택한 이유:
- 실제 브라우저 환경에서 테스트 가능
- 라우팅, 상태 렌더링, 버튼 상태, 접근성 역할까지 함께 검증 가능
- 티켓팅처럼 사용자 흐름이 중요한 UI에 적합함

## 5-2. Backend
- Go
- Gin
- GORM

### 왜 Go를 사용했는가
이 프로젝트의 핵심은 단순 비즈니스 로직 구현보다 **명확한 흐름 제어와 동시성 환경 대응**이다.  
Go는 다음 장점이 있었다.

- 단순하고 명확한 문법
- API 서버 구현에 적합
- 동시성 테스트를 goroutine으로 구성하기 좋음
- 학습과 설명 측면에서 구조가 직관적임

### 왜 Gin을 사용했는가
Gin은 Go 웹 프레임워크 중에서:
- 라우팅이 단순함
- REST API 구현이 빠름
- 학습 곡선이 비교적 낮음

즉 이 프로젝트의 초점인 **도메인 문제 해결**에 집중하기 좋았다.

### 왜 GORM을 사용했는가
직접 SQL만으로도 가능하지만,  
프로젝트 전체 구조를 빠르게 정리하고 모델 중심으로 관리하기 위해 GORM을 사용했다.

다만 중요한 지점에서는 ORM 추상화만 믿지 않고,
- transaction
- row lock
- unique constraint

처럼 **정합성 핵심 지점은 DB 중심 사고**로 설계했다.

---

## 5-3. Database / Cache
- PostgreSQL
- Redis

### 왜 PostgreSQL을 사용했는가
이 프로젝트의 핵심은 최종 예매 확정 시 데이터 정합성 보장이다.

PostgreSQL을 선택한 이유:
- transaction 지원이 강력함
- row-level lock 사용 가능
- unique constraint를 통한 최종 방어선 구성 가능
- 관계형 데이터 모델과 잘 맞음

즉 PostgreSQL은 “저장소”가 아니라  
**최종 확정과 정합성을 책임지는 시스템**으로 사용했다.

### 왜 Redis를 사용했는가
좌석 홀드는 최종 확정이 아니라 **짧은 시간 동안의 임시 선점**이다.  
이 단계는 빠른 응답이 중요하고 TTL 관리가 핵심이다.

Redis를 선택한 이유:
- `SETNX`로 중복 점유 방지
- TTL로 임시 홀드 만료 처리
- 빠른 응답 속도

즉 Redis는 **임시 홀드**, PostgreSQL은 **최종 확정** 역할로 분리했다.

## 6. 아키텍처 관점 핵심 설계

## 6-1. 백엔드 설계 핵심
이 프로젝트에서 백엔드의 핵심 질문은 이것이었다.

> 동일 좌석에 대한 동시 요청이 들어와도 최종 예매가 1건만 성공하도록 만들 수 있는가?

이를 위해 구조를 다음처럼 나눴다.

### 1) 좌석 홀드 단계
- Redis `SETNX + TTL`
- 빠른 임시 선점
- 중복 홀드 방지

### 2) 예매 확정 단계
- queue token 검증
- Redis hold 상태 확인
- PostgreSQL transaction 시작
- `SELECT ... FOR UPDATE`로 seat row lock 획득
- seat 상태 재검증
- booking row 생성
- queue 상태 갱신
- commit 성공 이후 Redis hold 제거

즉 예매 확정 전체를 하나의 transaction 단위로 묶어  
중간 실패 시 rollback 가능하도록 설계했다.

### 왜 이렇게 나눴는가
Redis 하나로 끝내면 빠르긴 하지만 최종 정합성을 강하게 설명하기 어렵고,  
DB만으로 홀드까지 처리하면 임시 선점 단계가 무거워진다.

따라서 역할을 분리했다.

- Redis = 빠른 임시 점유
- DB = 최종 확정과 정합성

---

## 6-2. 프론트 설계 핵심
프론트의 핵심 질문은 이것이었다.

> 서버 상태가 빠르게 바뀌는 도메인에서, 사용자가 믿을 수 있는 화면을 어떻게 유지할 것인가?

이 질문에 대해 다음을 설계했다.

### 1) 중복 행동 방지
- 대기열 진입 중복 호출 방지
- 좌석 홀드 버튼 연타 방지
- 예매 확정 버튼 연타 방지
- 요청 중 버튼 disabled
- 요청 중 상태 문구 노출

이유:
사용자가 “버튼을 여러 번 눌렀는데 중복 요청이 날아갔나?”를 걱정하지 않게 해야 하기 때문이다.

### 2) 화면 정합성 유지
예매 도메인에서는 optimistic update가 매우 위험할 수 있다.

예:
- 화면에는 AVAILABLE
- 실제 서버는 이미 HELD 또는 BOOKED

그래서 이 프로젝트에서는:
- optimistic update 최소화
- hold 성공 후 좌석 재조회
- hold 실패 후 좌석 재조회
- booking 성공 후 좌석 재조회
- booking 실패 후 좌석 재조회

를 적용했다.

즉 프론트는 “빠르게 보여주기”보다  
**거짓말하지 않는 화면**을 우선했다.

### 3) 렌더링 전략 분리
페이지 성격에 따라 렌더링 전략을 분리했다.

#### `/events`, `/events/[id]`
- 조회 중심
- 서버 중심 렌더링

이유:
- 공연 목록/상세는 비교적 정적
- 첫 화면 노출과 SEO가 중요
- 실시간 상호작용보다 조회 성격이 강함

#### `/queue/[eventId]`, `/booking/[eventId]`
- 실시간 상태 변화
- 사용자 상호작용 중심
- 클라이언트 중심 렌더링

이유:
- polling 필요
- seat selection / hold / booking 액션 존재
- queue token / storage / browser state 사용

### 4) 상태 저장 전략
프론트 저장 전략도 역할에 따라 분리했다.

#### `client_id` → localStorage
이유:
- 브라우저 단위 식별 보조값
- 새로고침 후에도 흐름 유지
- 로그인 없는 MVP 구조에서 최소한의 식별 역할

단, 보안 토큰이 아니라 **UX용 보조 식별값**으로만 취급했다.

#### `queue_token` → sessionStorage
이유:
- 예매 흐름용 세션성 데이터
- 장기 저장 불필요
- 세션 종료/완료 시 정리 용이
- query param보다 덜 노출됨

즉 프론트는 값을 저장하되, **절대 신뢰하지는 않는다**는 원칙을 지켰다.

### 5) 네트워크 실패 대응
모든 요청에 retry를 걸지 않았다.

#### retry 적용
- `fetchSeats`
- `getQueueStatus`

이유:
조회성 요청은 제한적 재시도가 UX에 도움 된다.

#### retry 미적용
- `enterQueue`
- `holdSeat`
- `createBooking`

이유:
상태 변경 요청, 특히 booking은 자동 재시도 시 사용자 신뢰를 해칠 수 있다.

예매 확정에서 중요한 것은:
- 자동 retry가 아니라
- 중복 요청 방지
- 명확한 결과 표시
- 서버 기준 재동기화

였다.

## 7. React Query 도입 배경과 결과

초기에는 로컬 state + fetch wrapper로 충분했지만,  
이후 프론트 메인 포트폴리오 관점에서 다음 질문이 생겼다.

> 서버 상태를 정말 서버 상태답게 다루고 있는가?

그래서 React Query를 도입했다.

### 왜 queue와 seats부터 적용했는가
이 둘이 가장 서버 상태 성격이 강했다.

#### queue status
- polling 필요
- `WAITING -> READY -> EXPIRED` 전환
- stale state 관리 중요

#### seats list
- hold / booking 이후 재조회 필요
- 서버 상태 변화가 잦음
- invalidate 기반 동기화가 자연스러움

### 도입 결과
- queue polling을 React Query 기준으로 관리
- READY / EXPIRED 시 polling 자동 중단
- seats list를 query로 관리
- hold / booking 후 `invalidateQueries`로 재동기화
- 페이지 내부에서 직접 fetch + setState 반복 감소

즉 문서화 수준이 아니라  
**실제로 서버 상태 관리 책임 일부를 React Query로 이동**했다는 점이 중요하다.

## 8. 성능 최적화 관점 정리

현재 좌석 수는 많지 않지만, 티켓팅 UI는 좌석 수가 늘어날수록 프론트 성능 문제가 생길 수 있다.

이 프로젝트에서는 지금 단계에서 의미 있는 최적화를 먼저 적용했다.

### 적용한 최적화
- `SeatSectionGrid` memoization
- section / seat card 단위 memoization
- section grouping `useMemo`
- `SeatSelectionSummary` memoization
- `useCallback`으로 stable callback 유지
- 파생값 `useMemo` 관리

### 왜 이 최적화를 먼저 했는가
아직 좌석 수가 500석, 1000석 수준은 아니기 때문에  
지금 당장 virtualization까지 넣는 것은 과할 수 있다.

대신:
- 불필요한 re-render 감소
- render churn 감소
- refetch 이후 하위 카드 흔들림 완화

같이 **지금 구조에서 효과가 큰 최적화**를 먼저 택했다.

### 왜 virtualization은 아직 안 했는가
현재 프로젝트 범위에서는:
- 좌석 수가 매우 많지 않음
- UX보다 과설계가 될 수 있음

다만 향후:
- 좌석 수가 수백 개 이상
- 스크롤 성능 저하
- 큰 구역별 렌더링 비용 증가

같은 상황이면 도입 후보가 된다.

---

## 9. 접근성 설계 관점 정리

접근성은 “aria 몇 개 붙였다”로 끝나는 게 아니라,  
**상태 변화가 많은 UI에서 의미와 포커스를 어떻게 전달할 것인가**가 중요하다.

### 기본 접근성 보강
- 에러 배너 `role="alert"`
- 상태 문구 `aria-live`
- 좌석 버튼 `aria-label`
- `aria-pressed`
- `aria-disabled`

### 더 깊은 접근성 보강
- 선택된 좌석으로 focus 이동
- 예매 완료 후 결과 카드 heading으로 focus 이동
- landmark 구조 정리
  - `section`
  - `aside`
  - `aria-labelledby`
- 선택 상태를 더 명확히 전달

### 왜 이렇게 했는가
티켓팅 UI는:
- 선택
- 대기
- 실패
- 완료

처럼 상태 전환이 많다.  
이때 시각적 변화만으로는 충분하지 않다.

그래서:
- 스크린리더가 상태를 읽을 수 있어야 하고
- 키보드 사용자도 현재 컨텍스트를 잃지 않아야 하며
- 예매 완료 같은 중요한 순간에 포커스가 적절히 이동해야 한다

는 점을 기준으로 접근성을 설계했다.

## 10. 테스트 전략

## 10-1. 백엔드 테스트
Go 테스트로 다음을 검증했다.

- 정상 예매 흐름
- 예매 오픈 전 진입 차단
- 이미 예매된 좌석 차단
- 홀드 없이 예매 차단
- 잘못된 queue token 차단
- 이미 홀드된 좌석 차단
- 동일 좌석 동시 hold 경쟁 제어
- 동일 좌석 동시 booking 경쟁 제어

### 왜 중요한가
백엔드는 동작만 하는 것보다  
**동시 요청 경쟁에서도 결과가 안전한지**가 더 중요하기 때문이다.

---

## 10-2. 프론트 테스트
Playwright 기반 E2E / smoke 테스트를 추가했다.

### 검증한 항목
- 공연 목록 렌더링
- 상세 이동
- queue 화면 상태 확인
- booking 직접 진입 차단
- 잘못된 진입 시 버튼 disabled 확인
- alert / live region / 버튼 노출 확인

### 왜 중요한가
프론트는 화면이 보여도,
- 라우팅
- 상태 렌더링
- 접근성 역할
- 잘못된 진입 차단

이 실제 브라우저에서 깨질 수 있다.

즉 프론트도 “수동으로 됐다”가 아니라  
**자동화된 브라우저 검증이 있다**는 점이 중요하다.

### 테스트 안정화에서 배운 점
초기에는 `getByRole("alert")`가 Next 내부 route announcer와 충돌했다.  
그래서:

- `error-banner`
- `loading-state`

에 `data-testid`를 추가하고,  
테스트 selector를 안정화했다.

즉 테스트도 그냥 만드는 것이 아니라  
**덜 깨지는 selector 전략까지 설계**했다.

---

## 10-3. 부하 테스트
k6를 사용해 주요 API의 부하와 경쟁 상황을 검증했다.

### queue enter
- burst traffic 시나리오
- 실패율 0%
- p95 응답시간: 175.81ms, 362.66ms

### seat hold / booking 경쟁
이 테스트는 모든 요청 성공이 목적이 아니라,  
**동일 좌석 경쟁 시 1건만 성공하고 나머지가 정상 거절되는지**를 보는 테스트다.

#### 확인 결과
- `seat_hold`: p95 16.22ms
- `booking`: p95 32.97ms

즉 경쟁 시나리오에서도
- 응답시간은 안정적이었고
- 일부 요청만 성공
- 나머지는 정상 거절

되는 구조를 확인했다.

## 11. 주요 트러블슈팅

## 11-1. Redis 홀드만으로는 최종 정합성을 보장하기 어려웠던 문제
### 문제
초기 구조에서는 좌석 홀드를 Redis로 빠르게 처리했지만,  
최종 예매 확정까지 Redis 중심으로만 처리하면 동일 좌석 경쟁 상황에서 정합성을 강하게 보장하기 어려웠다.

### 왜 어려웠는가
티켓팅에서는 임시 선점과 최종 확정의 성격이 다르다.  
홀드는 빠른 응답이 중요하지만, 최종 확정은 반드시 데이터 일관성이 더 중요하다.

### 해결
- 좌석 홀드: Redis `SETNX + TTL`
- 예매 확정: PostgreSQL transaction
- 경쟁 제어: `FOR UPDATE`
- DB 제약조건 추가

### 결과
- 동일 좌석 동시 hold 요청: 1건만 성공
- 동일 좌석 동시 booking 요청: 1건만 성공
- booking row도 최종 1건만 생성

---

## 11-2. 정상 흐름 구현만으로는 티켓팅 핵심 문제를 해결했다고 보기 어려웠던 문제
### 문제
정상 예매 흐름이 동작하는 것만으로는 티켓팅 핵심 문제를 해결했다고 보기 어려웠다.

### 해결
동시 hold / 동시 booking 시나리오를 테스트로 추가했다.

### 결과
동일 좌석에 대해 최종 성공 1건만 남는 것을 검증했다.

---

## 11-3. 프론트에서 stale state가 쉽게 생기는 문제
### 문제
좌석 상태가 빠르게 바뀌므로, 화면 상태와 서버 상태가 쉽게 어긋났다.

### 해결
- optimistic update 최소화
- 액션 후 재조회
- React Query invalidate 적용

### 결과
좌석 상태를 서버 기준으로 재동기화하는 구조를 만들었다.

---

## 11-4. 비동기 UI가 쉽게 불안정해지는 문제
### 문제
대기열은 polling, 좌석 선택은 재조회가 많아 중복 요청·stale response·잘못된 직접 진입 문제가 생기기 쉬웠다.

### 해결
- 중복 요청 방지
- polling 종료 조건 명확화
- 잘못된 직접 진입 차단
- query 기반 상태 관리

### 결과
queue status는 상태에 따라 polling이 자동 중단되고, booking 직접 진입은 UI 차원에서도 막을 수 있게 됐다.

---

## 11-5. k6 경쟁 테스트에서 실패율이 높아 보이는 문제
### 문제
경쟁 시나리오에서는 4xx가 많아 실패처럼 보일 수 있었다.

### 해결
일반 부하 테스트와 경쟁 제어 테스트의 해석 기준을 분리했다.

### 결과
- queue enter: 실패율 0%
- seat_hold: p95 16.22ms
- booking: p95 32.97ms

즉 “실패율”이 아니라 **정상 거절**로 해석해야 하는 시나리오임을 정리했다.

---

## 11-6. 프론트 테스트가 UI는 맞는데 깨지는 문제
### 문제
Playwright의 role selector가 Next 내부 요소와 충돌했다.

### 해결
- `data-testid` 추가
- stable selector 전략 사용

### 결과
직접 진입 차단, 버튼 disabled, loading / error / queue content 노출 등의 테스트를 더 안정적으로 검증할 수 있게 됐다.

---

## 12. 성과 정리

이 프로젝트를 통해 얻은 핵심 성과는 다음과 같다.

### 백엔드 관점
- Redis와 PostgreSQL의 역할을 분리해 설명 가능한 구조를 만들었다.
- transaction과 row-level lock을 실제 예매 흐름에 적용했다.
- 동시성 테스트로 “최종 1건 성공”을 검증했다.

### 프론트 관점
- 단순 UI 구현을 넘어 화면 정합성 문제를 설계했다.
- SSR/CSR 전략을 페이지 성격에 맞게 분리했다.
- React Query를 실제로 도입해 서버 상태를 관리했다.
- retry 정책을 요청 성격별로 다르게 설계했다.
- 접근성과 테스트 안정성까지 설명 가능한 수준으로 끌어올렸다.

### 검증 관점
- Go test
- Playwright
- k6

를 통해 기능, 브라우저 흐름, 부하/경쟁 상황을 각각 अलगভাবে 검증했다.

## 13. 자소서/면접에서 강조할 핵심 문장

### 프로젝트 소개 한 줄
대기열 기반 티켓 예매 시스템을 구현하면서, 백엔드에서는 동일 좌석 동시 요청에 대한 정합성을, 프론트에서는 실시간 상태 변화 속 화면 정합성과 사용자 신뢰를 중심으로 설계했습니다.

### 백엔드 핵심 강조 문장
좌석 홀드는 Redis `SETNX + TTL`로 처리하고, 최종 예매는 PostgreSQL transaction과 row-level lock으로 처리해 동일 좌석에 대한 중복 예매를 방지했습니다.

### 프론트 핵심 강조 문장
프론트는 단순 화면 구현보다 stale state, 중복 행동, polling, retry 정책, 접근성까지 포함해 실시간 예매 흐름에서 신뢰 가능한 UI를 만드는 데 초점을 맞췄습니다.

### 테스트 강조 문장
기능 구현만으로 끝내지 않고, Go 테스트로 동시성 시나리오를, Playwright로 실제 브라우저 흐름을, k6로 부하와 경쟁 상황을 검증했습니다.

## 14. 이 프로젝트가 신입 포트폴리오에서 가지는 의미

이 프로젝트는 단순히 여러 기술을 나열한 포트폴리오가 아니라,  
**문제를 정의하고, 기술을 목적에 맞게 선택하고, 검증까지 연결한 프로젝트**라는 점에서 의미가 있다.

특히 신입 관점에서 다음을 설명할 수 있다는 점이 강점이다.

- 왜 Redis와 PostgreSQL을 같이 썼는가
- 왜 lock과 transaction이 필요한가
- 왜 프론트에서 optimistic update를 최소화했는가
- 왜 queue는 CSR이고 events는 서버 중심 렌더링인가
- 왜 booking은 retry하지 않았는가
- 왜 React Query를 queue/seats부터 도입했는가
- 왜 접근성과 Playwright까지 넣었는가

즉 이 프로젝트는  
“기능을 구현해봤다” 수준이 아니라  
**설계 이유를 설명할 수 있는 포트폴리오**로 사용 가능하다.
